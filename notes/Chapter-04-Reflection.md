# 第 4 章：反思（Reflection）

> **一句话理解**：让 Agent 学会"回头看" — 生成输出后自我审查、发现问题、迭代改进，直到满意为止。这是从"执行者"迈向"思考者"的关键一步。

---

## 为什么需要它？

前三章的 Agent 都有一个共同问题：**不管输出质量如何，直接就是最终结果**。就像一个从不检查作业的学生 — 能写对当然好，写错了也交上去。

现实中，高质量的输出几乎都来自反复修改：
- 作家写初稿 → 审稿 → 改稿 → 再审 → 再改
- 程序员写代码 → 测试 → 修 bug → 再测 → 再修
- 设计师出方案 → 评审 → 修改 → 再评审

反思模式就是**把这个"改稿"过程自动化**。

---

## 核心思路：生成 → 评估 → 优化 → 循环

```
初始输入
   ↓
[生成] → 第一版输出
   ↓
[评估] → "这里有问题：A、B、C"
   ↓
[优化] → 第二版输出（修复了 A、B、C）
   ↓
[评估] → "还有问题 D" 或 "CODE_IS_PERFECT"
   ↓
  ...循环直到满意或达到最大次数
```

---

## 两种实现方式

### 方式 1：自我反思（单智能体）

同一个 Agent 既当作者又当审稿人：

```
Agent: 生成代码 → 自己审查 → 自己修改
```

- 优点：简单
- 缺点：有**认知偏差** — 自己写的东西自己往往看不出问题

### 方式 2：生产者-评审者（双智能体） ← 推荐

两个独立角色，各司其职：

```
生产者（如"Python开发者"）: 专注生成代码
   ↓ 输出
评审者（如"高级软件工程师"）: 专注找问题、给反馈
   ↓ 反馈
生产者: 根据反馈修改
   ↓ ...循环
```

为什么更好？**关注点分离** — 评审者以"全新视角"看输出，不会对自己的作品手下留情。

---

## 6 大应用场景

| 场景 | 反思做什么 | 效果 |
|------|-----------|------|
| **创意写作** | 评审流畅性/语气/清晰度 → 重写 | 更精炼有效的内容 |
| **代码生成** | 运行测试/静态分析 → 修复 | 更健壮实用的代码 |
| **复杂推理** | 评估中间步骤是否合理 → 回溯 | 更好地导航问题空间 |
| **摘要优化** | 与原文关键点对比 → 补充 | 更准确全面的摘要 |
| **规划评估** | 模拟执行/可行性检查 → 修订 | 更现实有效的计划 |
| **对话审查** | 检查对话连贯性 → 纠正 | 更自然的多轮对话 |

## 代码示例要点

### LangChain 方式 — 手动迭代循环

```python
for i in range(max_iterations):
    # 1. 生成/优化
    if i == 0:
        response = llm.invoke(message_history)  # 首次生成
    else:
        response = llm.invoke(message_history)  # 根据评审修改

    # 2. 评审（用不同角色的系统提示词）
    reflector_prompt = [
        SystemMessage(content="你是高级软件工程师，执行细致的代码审查..."),
        HumanMessage(content=f"要审查的代码：\n{current_code}")
    ]
    critique = llm.invoke(reflector_prompt)

    # 3. 停止条件
    if "CODE_IS_PERFECT" in critique:
        break  # 满意了，退出循环

    # 把评审加入历史，下一轮继续优化
    message_history.append(critique)
```

关键点：
- **对话历史贯穿全程** — 每轮评审和修改都在累积的上下文中进行
- **明确的停止条件** — "CODE_IS_PERFECT" 或达到最大迭代次数
- **角色切换** — 生成和评审用不同的系统提示词

### Google ADK 方式 — 顺序智能体

```python
generator = LlmAgent(name="DraftWriter", output_key="draft_text")
reviewer  = LlmAgent(name="FactChecker", output_key="review_output")

pipeline = SequentialAgent(sub_agents=[generator, reviewer])
# 生成器写入 state → 评审者从 state 读取 → 输出评审结果
```

关键点：通过 `output_key` 共享状态，评审者自动获取生成者的输出。

## 重要权衡

| 收益 | 代价 |
|------|------|
| 输出质量显著提升 | 每轮迭代都要额外 LLM 调用（**成本 ↑**） |
| 自我纠错能力 | 迭代过程增加延迟（**速度 ↓**） |
| 更准确、连贯的结果 | 对话历史越来越长（**可能超出上下文窗口**） |

**经验法则**：质量 > 速度时用反思；速度 > 质量时跳过。

---

## 本质理解

反思的本质是**引入反馈循环（feedback loop）**。前三章的 Agent 都是"开环系统"（open-loop）— 执行完就结束。反思让它变成"闭环系统"（closed-loop）— 输出会被反馈回来作为下一轮的输入。

类比：
- 开环 = 导弹发射后不再调整轨迹
- 闭环 = 导弹根据目标位置持续修正轨迹（制导系统）

反思就是 Agent 的"制导系统"。

```
反思 → 多智能体（Ch.7）= "批评者-审查者"模式是多智能体的雏形
```

## 速览（At a Glance）

- **问题背景**：Agent 初始输出往往次优，缺乏识别和修复自身错误的内置流程
- **解决方案**：建立"生成→评估→优化"反馈循环；推荐用独立评审者避免认知偏差
- **实践建议**：当质量/准确性比速度/成本更重要时使用；适合长篇内容、代码调试、详细计划等任务
