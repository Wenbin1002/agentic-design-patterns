# 第 6 章：规划（Planning）

> **一句话理解**：Agent 不再按预设步骤执行，而是**自己动态生成执行计划** — 接到目标后，自主分解为子步骤、安排顺序、调用工具，必要时还能调整计划。从"执行者"升级为"策略家"。

---

## 为什么需要它？

提示词链（第 1 章）的步骤是**你预先设计好的** — 你告诉 Agent "第一步做 X，第二步做 Y"。但很多现实问题：

- 你只知道**目标**，不知道具体怎么做
- 步骤数量和内容**因输入不同而变化**
- 执行过程中可能需要**动态调整**

例如：
```
用户: "帮我写一份关于 AI 在医疗领域应用的研究报告"

预设链无法处理 — 你不知道该搜什么、搜多少、怎么组织
规划 Agent → 自主生成计划：
  1. 搜索"AI 医疗 最新进展"
  2. 搜索"AI 诊断 案例"
  3. 搜索"AI 药物研发"
  4. 综合各来源信息
  5. 组织为报告结构
  6. 撰写并润色
```

---

## 核心区分：动态规划 vs 固定工作流

这是本章最重要的决策框架：

| | 动态规划 | 固定工作流 |
|--|---------|----------|
| **适用条件** | 「方法」需要被发现 | 「方法」已经已知 |
| **步骤来源** | Agent 根据输入动态生成 | 开发者预先定义 |
| **灵活性** | 高 — 可适应新信息 | 低 — 固定路径 |
| **可预测性** | 低 — 可能走偏 | 高 — 确定性结果 |
| **典型场景** | 研究报告、竞争分析 | 订单处理、入职流程 |

**经验法则**：如果你能写出明确的流程图，就用固定工作流；如果你写不出来，就用规划。

---

## 规划的工作流程

```
用户: "关于 X 的研究报告"
    ↓
① [理解目标] → 明确初始状态和目标状态
    ↓
② [生成计划] → LLM 分解为子步骤序列
    ↓
③ [呈现计划] → （可选）让用户审阅/修改
    ↓
④ [执行计划] → 逐步执行，调用工具
    ↓
⑤ [适应调整] → 遇到问题时重新规划
    ↓
⑥ [综合输出] → 整合所有步骤的结果
```

## 两个标杆案例：Deep Research

### Google DeepResearch

这是规划模式的**商业化标杆**：

```
用户提问 → 生成多点研究计划 → 用户审阅/修改计划
    ↓
迭代搜索-分析循环：
  搜索 → 分析 → 发现知识缺口 → 新搜索 → 分析 → ...
    ↓
综合阶段：批判性评估 → 组织结构 → 生成带引用的多页报告
```

关键特征：
- **计划是动态的** — 根据搜到的信息调整后续搜索
- **人机协作** — 计划生成后让用户审阅再执行
- **异步长时运行** — 可能分析数百个来源，用户可以先走
- **带引用** — 每个结论都链接到原始来源

### OpenAI Deep Research API

```python
response = client.responses.create(
    model="o3-deep-research-2025-06-26",
    input=[system_message, user_query],
    tools=[{"type": "web_search_preview"}]
)
# API 暴露所有中间步骤：推理、搜索查询、代码执行
```

关键特征：
- **透明度** — 暴露推理步骤、搜索查询、代码执行
- **可扩展** — 支持 MCP 连接私有数据源
- **结构化输出** — 带内联引用和元数据

## 代码示例要点

### CrewAI 方式 — 最简规划

```python
agent = Agent(
    role='文章规划者和撰写者',
    goal='规划然后撰写关于指定主题的摘要。',
)

task = Task(
    description=(
        "1. 为主题创建要点计划。\n"
        "2. 根据计划撰写摘要。"
    ),
    expected_output="### 计划\n...\n### 摘要\n..."
)
```

关键点：**在 task description 里明确要求"先计划再执行"** — 通过提示词引导 LLM 展现规划行为。

## 规划与提示词链的本质区别

```
提示词链（Ch.1）：
  开发者预定义步骤 → Agent 按顺序执行
  A → B → C → D（固定的）

规划（Ch.6）：
  开发者给目标 → Agent 自己生成步骤 → 执行 → 可能调整
  目标 → [LLM 生成: A → B → C] → 执行 → 发现问题 → [重新规划: A → B → D → E]
```

| | 提示词链 | 规划 |
|--|---------|------|
| 步骤来源 | 人工设计 | LLM 生成 |
| 是否固定 | 固定不变 | 可动态调整 |
| 控制力 | 高 | 低 |
| 灵活性 | 低 | 高 |

---

## 本质理解

规划的本质是**让 LLM 充当"项目经理"** — 你给它一个目标，它自己拆解任务、安排执行顺序、分配资源（工具），遇到问题还能调整方案。

类比：
- 提示词链 = 按菜谱做菜（步骤固定）
- 规划 = 给厨师一个"做一桌中餐"的目标，让他自己决定做什么菜、用什么食材、怎么安排时间

---

## 与其他章节的关系

```
规划 是提示词链的"高级形态"：
  提示词链 = 预定义的步骤序列
  规划 = 动态生成的步骤序列

规划 + 工具使用（Ch.5）= 计划中每步可能涉及工具调用
规划 + 反思（Ch.4）    = 执行计划后评估效果，调整计划
规划 + 多智能体（Ch.7）= 将计划的不同步骤分配给不同 Agent
规划 + 目标监控（Ch.11）= 跟踪计划执行进度，偏离时重新规划
```

## 实践中的核心 Tradeoff（书中未深入）

### Tradeoff 1：计划质量 vs 执行可靠性

LLM 生成的计划可能忽略依赖关系、包含不可执行步骤、或粒度不对。

应对策略：

| 策略 | 做法 | 代价 |
|------|------|------|
| **验证门控** | 每步执行后验证结果，失败则重新规划 | 延迟 ↑↑ |
| **降低自主度** | 限制可用工具集，约束计划空间 | 灵活性 ↓ |
| **人工审批** | 计划生成后让人确认再执行 | 不能全自动 |
| **分层规划** | 粗计划 → 执行一步 → 根据结果细化下一步 | 复杂度 ↑ |

### Tradeoff 2：规划深度 vs 计算成本

```
Plan-then-Execute（一次性全量规划）：
  快，但后续步骤基于假设，可能脱离实际

Plan-as-you-go（逐步规划 / ReAct）：
  每步基于真实反馈，但每步都要一次 LLM 调用，慢且贵

混合策略（实践中最常用）：
  先粗粒度规划全局 → 执行一步 → 根据结果微调后续 → 局部重新规划
```

### Tradeoff 3：Agent 失控风险

Agent 可能进入无限循环或执行危险操作。防御措施：

```python
MAX_STEPS = 10           # 步骤上限
MAX_LLM_CALLS = 20      # LLM 调用上限
MAX_TIME = 300           # 时间上限（秒）
ALLOWED_TOOLS = [...]    # 操作白名单（不给危险工具）
CHECKPOINT_EVERY = 3     # 每 N 步人工确认
```

### Tradeoff 4：计划表示格式

| 格式 | 优点 | 缺点 |
|------|------|------|
| 自然语言列表 | 生成容易 | 解析困难，边界模糊 |
| 结构化 JSON | 精确可执行 | LLM 可能生成无效格式 |
| 函数调用序列 | 最精确 | 灵活性低 |
| DAG（有向无环图） | 支持并行执行 | 生成和验证复杂 |

最佳实践 — 半结构化格式：自然语言描述 + 结构化元数据（含 success_criteria 和 fallback）。

### 什么时候不该用规划？

| 场景 | 原因 | 更好的选择 |
|------|------|------|
| 流程已标准化 | 引入不必要的不确定性 | 固定工作流 |
| 要求 100% 确定性 | LLM 规划不可控 | 硬编码逻辑 |
| 延迟敏感（< 2s） | 规划本身要花时间 | 预计算/缓存 |
| 步骤少（1-3步） | 杀鸡用牛刀 | 简单链式调用 |
| 安全关键操作 | Agent 可能做危险决策 | 人工审批流 |

### 成熟规划系统需要的 6 个要素

1. **分层规划**（粗到细，逐步细化）
2. **每步验证 + 失败回退**（不盲目执行下一步）
3. **硬性资源限制**（时间/token/步数/成本）
4. **操作白名单**（防止 Agent 失控）
5. **人工检查点**（高风险时暂停确认）
6. **结构化计划格式**（可解析、可验证、含回退策略）

> 这些能力分散在后续章节：第 11 章目标监控、第 12 章异常处理、第 13 章人机协同。

## 速览（At a Glance）

- **问题背景**：复杂问题无法单步解决，需要远见卓识和多步骤相互依赖的操作
- **解决方案**：让 Agent 先生成连贯计划（分解目标为子步骤），再按计划执行，必要时动态调整
- **实践建议**：当请求太复杂无法单步处理时使用；关键决策 —「方法需要被发现」→ 用规划；「方法已知」→ 用固定工作流
