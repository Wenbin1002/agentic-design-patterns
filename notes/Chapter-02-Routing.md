# 第 2 章：路由（Routing）

> **一句话理解**：给 Agent 加上"if-else"能力 — 根据输入的类型/意图，动态选择不同的处理路径，把固定流水线变成自适应决策系统。

---

## 为什么需要它？

第 1 章的提示词链是**固定的线性流程** — A → B → C，无论输入是什么，都走同一条路。但现实世界的输入千变万化：

```
用户A: "帮我订去伦敦的机票"    → 应该走预订流程
用户B: "意大利的首都是什么？"   → 应该走信息检索
用户C: "我的订单到哪了？"       → 应该走数据库查询
用户D: "（一堆乱码）"          → 应该走澄清流程
```

没有路由，你只能把所有逻辑塞进一个提示词里，或者用固定的链处理所有情况 — 要么低效，要么失败。

---

## 核心思路：分类 → 分发

路由的本质就两步：

```
第1步：判断输入属于哪一类（分类）
第2步：把请求转发给对应的专门处理器（分发）
```

用客户支持场景理解：

```
用户查询 → [路由器] → 判断意图
                ├→ "检查订单"  → 订单数据库查询智能体
                ├→ "产品咨询"  → 产品目录搜索智能体
                ├→ "技术支持"  → 故障排除智能体 / 升级人工
                └→ "意图不清"  → 澄清子流程
```

---

## 4 种路由实现方式

| 方式 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **基于 LLM** | 提示 LLM 分析输入，输出类别标识 | 灵活，能理解语义 | 有延迟和成本，非确定性 |
| **基于嵌入** | 将输入转为向量，与各路由的向量做相似度对比 | 语义理解好 | 需要预计算路由嵌入 |
| **基于规则** | 关键词/正则/if-else 预定义逻辑 | 快、确定性强 | 灵活性差，无法处理新颖输入 |
| **基于 ML 模型** | 在标注数据上微调分类器 | 快速且准确 | 需要训练数据，不如 LLM 灵活 |

**实践选择**：
- 场景简单、追求速度 → 基于规则
- 输入多样、需要语义理解 → 基于 LLM 或嵌入
- 有历史数据、追求性价比 → 微调分类器

---

## 路由可以出现在哪里？

不只是入口处！路由可以在工作流的**多个节点**发挥作用：

```
[入口路由] → 对主任务分类，选择总体流程
      ↓
[中间路由] → 链的某一步完成后，决定下一步走哪条路
      ↓
[工具路由] → 子程序内部，从可用工具集中选最合适的
```

---

## 代码示例要点

### LangChain 方式 — 显式路由

```python
# 1. 用 LLM 做分类决策
coordinator_router_chain = coordinator_router_prompt | llm | StrOutputParser()
# → 输出 "booker" / "info" / "unclear"

# 2. 用 RunnableBranch 做条件分发
delegation_branch = RunnableBranch(
    (lambda x: x['decision'] == 'booker', booking_handler),
    (lambda x: x['decision'] == 'info', info_handler),
    unclear_handler  # 默认分支
)
```

关键点：**LLM 只负责分类判断，实际处理交给专门的处理函数** — 分离了决策和执行。

### Google ADK 方式 — 隐式路由

```python
coordinator = Agent(
    instruction="分析请求并委托给适当的专家智能体...",
    sub_agents=[booking_agent, info_agent]  # 定义子智能体
)
# ADK 自动根据指令和子智能体描述做路由（自动流）
```

关键点：**不需要手写路由逻辑**，框架根据智能体的 `description` 自动匹配 — 更简洁但控制力稍弱。

---

## 本质理解

路由的本质是**软件工程中"策略模式"（Strategy Pattern）在 AI 领域的应用** — 根据运行时条件选择不同的算法/策略来处理同一个请求。

也可以类比**网络路由器**：数据包进来，查路由表，转发到正确的端口。只不过这里的"路由表"可以是 LLM 的语义理解。

---

## 与其他章节的关系

```
第 1 章 提示词链（顺序）
    ↓ + 条件分支
第 2 章 路由（选择） ← 你在这里
    ↓ + 多路同时
第 3 章 并行化（并发）
```

- **提示词链 + 路由** = 有条件分支的工作流（最常见的组合）
- **路由 + 工具使用**（第 5 章）= 选择用哪个工具，路由的一种具体应用
- **路由 + 多智能体**（第 7 章）= 把请求分配给不同的专家智能体
- **路由 + 资源优化**（第 16 章）= 根据任务复杂度路由到不同规格的模型（便宜/贵）

---

## 速览（At a Glance）

- **问题背景**：单一线性流程无法处理多样化输入，缺乏上下文决策能力，系统僵化非自适应
- **解决方案**：分析输入确定意图 → 动态将控制流导向最合适的工具/函数/子智能体，可由 LLM、规则或嵌入驱动
- **实践建议**：当 Agent 必须根据输入在多个工作流/工具/子智能体之间做选择时使用（如客户支持区分不同类型查询）
