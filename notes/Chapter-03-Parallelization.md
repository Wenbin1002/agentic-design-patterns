# 第 3 章：并行化（Parallelization）

> **一句话理解**：识别工作流中互不依赖的任务，同时执行而不是排队等待 — 总耗时从"所有任务之和"降为"最慢那个任务的时间"。

---

## 为什么需要它？

前两章解决了"怎么做"（提示词链）和"做什么"（路由），但都是**一个接一个**地执行。当任务涉及多个独立的外部调用时，顺序执行是巨大的浪费：

```
顺序执行（总耗时 = 各任务之和）：
  搜索新闻 (2s) → 查股票 (1.5s) → 爬社交媒体 (3s) → 查数据库 (1s)
  总计：7.5 秒

并行执行（总耗时 = 最慢的那个）：
  搜索新闻 (2s)   ┐
  查股票 (1.5s)    ├→ 全部同时开始，等最慢的完成
  爬社交媒体 (3s)  │
  查数据库 (1s)    ┘
  总计：3 秒
```

**关键前提**：任务之间**没有数据依赖** — A 的结果不需要作为 B 的输入。

---

## 核心思路：分叉 → 并发 → 汇聚

```
输入
  ├→ 任务 A（独立执行）──┐
  ├→ 任务 B（独立执行）──┼→ 汇聚点 → 综合处理 → 最终输出
  └→ 任务 C（独立执行）──┘
```

这个模式几乎总是和**顺序执行结合使用**：
- **并行阶段**：收集/处理互不依赖的数据
- **顺序阶段**：综合并行结果、做最终决策

---

## 7 大应用场景

| 场景 | 并行任务示例 | 优势 |
|------|-------------|------|
| **信息收集** | 同时搜新闻、查股票、爬社交媒体、查数据库 | 快速获得全面视图 |
| **数据分析** | 同时做情感分析、关键词提取、分类、紧急识别 | 多维度同时出结果 |
| **多 API 调用** | 同时查航班、酒店、活动、餐厅 | 更快呈现完整方案 |
| **内容生成** | 同时生成标题、正文、图片、按钮文案 | 高效组装内容 |
| **并发验证** | 同时验证邮箱、手机、地址、内容安全 | 更快反馈有效性 |
| **多模态处理** | 同时分析文本情感 + 识别图像对象 | 跨模态快速整合 |
| **A/B 多选项** | 同时生成 3 个不同标题供选择 | 快速比较选最优 |

## 代码示例要点

### LangChain 方式 — `RunnableParallel`

```python
# 三个独立链同时执行
map_chain = RunnableParallel({
    "summary": summarize_chain,      # 总结
    "questions": questions_chain,     # 生成问题
    "key_terms": terms_chain,         # 提取术语
    "topic": RunnablePassthrough(),   # 透传原始输入
})

# 并行结果 → 综合提示词 → LLM → 最终输出
full_chain = map_chain | synthesis_prompt | llm | StrOutputParser()
```

关键点：`RunnableParallel` 把多个链打包，LCEL 运行时自动并发执行。

### Google ADK 方式 — `ParallelAgent` + `SequentialAgent`

```python
# 三个研究员同时工作
parallel_research = ParallelAgent(
    sub_agents=[researcher_1, researcher_2, researcher_3]
)

# 先并行研究，再顺序综合
pipeline = SequentialAgent(
    sub_agents=[parallel_research, merger_agent]
)
```

关键点：
- 每个研究员通过 `output_key` 把结果写入共享状态
- 合并智能体从状态中读取所有结果进行综合
- `SequentialAgent` 套 `ParallelAgent` = 最常见的组合

## 重要注意事项

### 并发 ≠ 并行

Python `asyncio` 提供的是**并发**（concurrency），不是真正的**并行**（parallelism）：
- 并发：单线程事件循环，在 I/O 等待时切换任务（适合网络请求）
- 并行：多核同时计算（需要多进程，受 GIL 限制）

对于 Agent 场景（大量 API 调用、LLM 请求），**并发就够了** — 瓶颈在 I/O 等待而非 CPU 计算。

### 权衡

- **优势**：显著减少延迟，特别是涉及外部服务时
- **代价**：增加调试和日志的复杂性，需要处理部分失败的情况

---

## 本质理解

并行化的本质是**识别任务依赖图中的独立节点，让它们同时执行**。

类比：一个厨师做菜，煮饭（15分钟）和炒菜（10分钟）可以同时进行，但"盛饭"必须等煮饭完成。好的厨师会识别哪些步骤可以同时做 — 这就是并行化。

---

## 三种基本控制流的关系

到这里，前三章构成了 Agent 系统的**三种基本控制流**：

```
第 1 章 提示词链 ── 顺序执行（A → B → C）
第 2 章 路由     ── 条件分支（if X → A, elif Y → B, else → C）
第 3 章 并行化   ── 并发执行（A & B & C → 汇聚）
```

这三种控制流可以**任意组合嵌套**：
- 并行 + 顺序：先并发收集数据，再顺序综合（最常见）
- 路由 + 并行：根据类型路由后，某些路径内部使用并行
- 链 + 路由 + 并行：复杂工作流的完整控制流

**后续所有高级模式都建立在这三种基础控制流之上。**

---

## 速览（At a Glance）

- **问题背景**：纯顺序执行效率低下，外部 I/O 操作成为延迟瓶颈，总处理时间等于所有任务之和
- **解决方案**：识别无数据依赖的组件并同时执行，框架提供内置并发支持（RunnableParallel / ParallelAgent）
- **实践建议**：当工作流包含多个可同时运行的独立操作时使用（多 API 获取、多数据块处理、多内容片段生成）
