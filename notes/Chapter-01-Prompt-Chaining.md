# 第 1 章：提示词链（Prompt Chaining）

> **一句话理解**：把一个大问题拆成多个小步骤，每步的输出喂给下一步 — 这就是所有 Agent 系统最基础的骨架。

---

## 为什么需要它？

单个复杂提示词会让 LLM 出现 **5 种典型故障**：

| 故障 | 表现 |
|------|------|
| **指令忽略** | 提示词太长，部分指令被无视 |
| **上下文偏离** | 写着写着"忘了"最初的目标 |
| **错误传播** | 前面错了，后面越错越离谱 |
| **窗口不足** | 信息太多塞不进上下文 |
| **幻觉** | 认知负荷太高，开始编造事实 |

---

## 核心思路：分而治之

用书中的市场报告例子来理解：

```
单一提示词："分析这份市场报告，总结发现，识别趋势，然后写一封邮件"
  → 总结可能不错，但趋势提取和邮件撰写大概率翻车

提示词链：
  步骤1（分析师角色）: "总结这份市场报告" → 得到摘要
  步骤2（分析师角色）: "从摘要中提取3个趋势 + 数据" → 得到结构化趋势
  步骤3（写作者角色）: "根据趋势写一封营销邮件" → 得到最终邮件
```

**关键技巧**：步骤之间用 **JSON 等结构化格式** 传递数据，而不是自然语言 — 避免歧义导致下一步出错。

---

## 7 大应用场景

1. **信息处理工作流** — 提取 → 总结 → 实体识别 → 查库 → 生成报告
2. **复杂查询回答** — 拆子问题 → 分别研究 → 综合答案（可结合并行处理）
3. **数据提取和转换** — 初次提取 → 格式验证 → 补充提取 → 循环验证
4. **内容生成工作流** — 构思 → 大纲 → 逐段写作 → 审查完善
5. **有状态的对话系统** — 处理话语 → 识别意图/实体 → 更新状态 → 生成回复
6. **代码生成和完善** — 理解需求 → 伪代码 → 编码 → 查错 → 重写 → 加文档/测试
7. **多模态和多步推理** — 提取图片文字 → 关联标签 → 结合表格解释

---

## 进阶概念：上下文工程

这章提出了一个重要概念 — **上下文工程（Context Engineering）**，比"提示工程"更进一步：

- **提示工程** = 优化你问问题的措辞
- **上下文工程** = 为 AI 构建完整的信息环境

上下文工程包括 4 层信息：

| 层级 | 说明 | 示例 |
|------|------|------|
| **系统提示词** | 定义角色和行为边界 | "你是一名技术作家，语气必须正式且精确" |
| **检索文档** | 从知识库主动拉取相关信息 | 项目的技术规格文档 |
| **工具输出** | API 返回的实时数据 | 查询日历获取用户可用时间 |
| **隐式数据** | 用户身份、历史、环境状态 | 与邮件收件人的关系、之前的会议记录 |

核心观点：即使是最先进的模型，如果给它的上下文贫乏或构建不当，表现也会很差。

## 代码示例要点

书中用 LangChain 演示了一个两步链：

```python
# 步骤 1：提取技术规格
extraction_chain = prompt_extract | llm | StrOutputParser()

# 步骤 2：将提取结果转为 JSON
full_chain = (
    {"specifications": extraction_chain}
    | prompt_transform
    | llm
    | StrOutputParser()
)
```

关键点：
- 使用 LangChain 表达式语言（LCEL）的 `|` 管道符连接各步骤
- `StrOutputParser()` 将 LLM 消息输出转为字符串，供下一步使用
- 前一个链的输出自动填入下一个提示词的变量中

---

## 本质理解

提示词链的本质是**软件工程中"管道"思想在 AI 领域的应用** — 就像 Unix 的 `cat file | grep pattern | sort | head`，每个命令只做一件事，通过管道串起来解决复杂问题。

## 与后续章节的关系

提示词链是**全书的地基**，后续所有模式都建立在"链式执行"之上：

```
提示词链（顺序执行）
  + 条件判断 → 第 2 章 路由
  + 并发执行 → 第 3 章 并行化
  + 自我审查循环 → 第 4 章 反思
  + 外部 API 调用 → 第 5 章 工具使用
  + 动态步骤生成 → 第 6 章 规划
  + 多个链协作 → 第 7 章 多智能体协作
```

---

## 速览（At a Glance）

- **问题背景**：复杂任务在单个提示词内处理会导致指令忽略、上下文偏离、幻觉等问题
- **解决方案**：分解为聚焦的顺序子任务，每步输出作为下步输入，支持步骤间集成外部工具和结构化数据
- **实践建议**：当任务涉及多个处理阶段、需要外部工具交互、或需要多步推理和状态维护时使用
